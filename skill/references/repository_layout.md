# Repository Layout

Reference for the file structure of a course repository. Used by all commands
that need to locate or create files.

---

## Course root

```
<course-root>/
  CLAUDE.md               ‚Üê this skill + course-specific context (includes Profile:)
  course_plan.md          ‚Üê master course plan (source of truth)
  syllabus.md             ‚Üê student-facing syllabus (generated from course_plan.md)
  COURSE_STATE.md         ‚Üê status table for all lectures/labs
  course_conventions.md   ‚Üê language rules + terminology (generated by course init)
  slides_preamble.tex     ‚Üê LaTeX/Beamer preamble (generated by course init)
  lab_templates.md        ‚Üê notebook/test templates (generated by lab course-init)
  lms_adapter.md          ‚Üê LMS publish workflow (copied from profile during lab course-init)
  lectures/               ‚Üê per-lecture artifacts
  seminars/               ‚Üê per-seminar artifacts; same layout as lectures/ when
                            a seminar has a prepared deck. What a "seminar" means
                            and how its practical part is run vary by course.
  labs/                   ‚Üê per-lab artifacts + shared/ templates
  quizzes/                ‚Üê per-quiz artifacts (tests / exams)
```

---

## Lectures

```
lectures/
  01/
    plan.md             ‚Üê detailed slide plan (Step 1 output)
    visuals.md          ‚Üê visualization list (Step 2 output)
    figures/
      figures.py        ‚Üê figure generation script (Step 3 output)
      fig01_name.png    ‚Üê generated figures
      ...
    slides.tex          ‚Üê Beamer presentation (Step 4 output)
    speaker_notes.md    ‚Üê speaker text (Step 5 output)
    history.md          ‚Üê decision log for this lecture
  02/
    ...
```

Every step output lives in `lectures/NN/`. `history.md` is the per-lecture
memory of past iterations ‚Äî always read it before starting any step.

---

## Seminars

```
seminars/
  01/
    plan.md, visuals.md, figures/, slides.tex, speaker_notes.md, history.md
                        ‚Üê deck: same artifacts as lectures/NN/ (the seminar
                          commands reuse the lecture step references)
    practice.ipynb      ‚Üê practical part: live code-demo notebook
                          (`/course-maker seminar practice N`)
  02/
    ...
```

A seminar is the deck (the lecture pipeline, retargeted to `seminars/NN/`) plus
a practical part. The form of the practical part varies by course; here it is a
Jupyter demo notebook and is not autograded (unlike a lab).

---

## Labs

```
labs/
  shared/
    tests_template.py   ‚Üê style reference for Stage 2 (never edit)
    conftest_base.py    ‚Üê base conftest, copied per-lab then scoring updated
    tests.yaml          ‚Üê GitHub Actions CI (never modify)
  lab1/                 ‚Üê or labs/lab1-backprop/ if a slug was provided
    lab_spec.md         ‚Üê Stage 1 output, instructor-only (not in starter repo)
    history.md          ‚Üê decision log, same role as lectures/NN/history.md
    starter/            ‚Üê git subtree ‚Üí public starter repo (LMS-bound; see lms_adapter.md)
      exercises.ipynb
      conftest.py
      tests.py
      requirements.txt
      README.md
      datasets_info.md  ‚Üê if needed
      .github/workflows/tests.yaml
  lab2/
    ...
```

**`<LAB_DIR>` resolution.** Every `/course-maker lab N` command starts with:
read `COURSE_STATE.md` Labs table, find the row where `#` = N, read the `Dir`
column. That value is `<LAB_DIR>` ‚Äî use it as `labs/<Dir>/`. If the `Dir`
column is absent or empty, fall back to `labs/labN/`.

**Instructor-only vs student-facing.** Files in `labs/labN/` (`lab_spec.md`,
`history.md`) are NOT published to students. Only `labs/labN/starter/` contents
are published according to the LMS workflow in `<course-root>/lms_adapter.md`.

---

## Quizzes

```
quizzes/
  01/
    quiz_plan.md        ‚Üê plan: header, type legend, structure (Step 1 output)
    quiz_questions.md   ‚Üê canonical question bank, answers inline (Step 2 output);
                          instructor-only ‚Äî doubles as the answer key
    quiz_student.md     ‚Üê exported student-facing version, answers stripped
                          (Step 3 output; or quiz_variant_<letter>.md per variant)
    history.md          ‚Üê decision log, same role as lectures/NN/history.md
  02/
    ...
```

**Instructor-only vs student-facing.** `quiz_questions.md`, `quiz_plan.md`, and
`history.md` are NOT given to students ‚Äî the bank holds the answers. Only the
exported `quiz_student.md` / `quiz_variant_*.md` are handed out.

```markdown
| # | Title | plan | visuals | figures | slides | notes | practice | Updated |
|---|-------|------|---------|---------|--------|-------|----------|---------|
| 01 | ... | ‚úÖ | ‚úÖ | ‚úÖ | ‚úÖ | ‚úÖ | üîÑ | 2026-03-15 |

## Labs

| # | Dir | Title | plan | notebook | spec | tests | validated | published | Updated |
|---|-----|-------|------|----------|------|-------|-----------|-----------|---------|
| 01 | lab1-backprop | Backpropagation | ‚ùå | ‚ùå | ‚ùå | ‚ùå | ‚ùå | ‚ùå | ‚Äî |

| 01 | Midterm | ‚ùå | ‚ùå | ‚ùå | ‚Äî |

Legend: ‚úÖ done ¬∑ üîÑ in progress ¬∑ ‚ùå not started ¬∑ ‚öÝÔ∏è needs review
```

**Structural vocabulary is always English, regardless of course language.**
Section headings (`## Lectures`, `## Seminars`, `## Labs`, `## Quizzes`), column
names (`plan`, `visuals`, `figures`, `slides`, `notes`, `notebook`, `spec`,
`tests`, `questions`, `practice`, `validated`, `published`, `#`, `Dir`, `Title`,
`Updated`),
and the status legend
are language-neutral keys. Only cell *content* (titles) is in the course
language. This keeps the file machine-readable across all courses ‚Äî the drift
checker (`scripts/validate_state.py`) matches these exact English keys. Do not
translate or merge them: use separate `## Lectures` and `## Seminars` sections
with English column headers, and put the course-language titles in the `Title`
column.

Mark a step ‚öÝÔ∏è (needs review) when:
- `course_plan.md` was updated and this lecture's content may be affected.
- The user manually edited a file without going through the pipeline.

### `history.md`

Per-lecture (`lectures/NN/history.md`) and per-lab (`<LAB_DIR>history.md`).

This is the memory of past iterations. **Always read it before starting any
step for that lecture/lab.** It prevents re-proposing ideas the user already
rejected.

Append after each completed step or meaningful iteration. Format:

```markdown
# Lecture N ‚Äî History

## [YYYY-MM-DD] Step 1: Plan ‚Äî iteration 1
**Result:** plan.md created
**User feedback:** "Too much time on section 3, compress to 1 slide"
**Decision:** Merged slides 8‚Äì10 into one overview slide

## [YYYY-MM-DD] Step 1: Plan ‚Äî iteration 2
**Result:** plan.md updated
**User feedback:** Approved

## [YYYY-MM-DD] Manual edit
**What changed:** User rewrote slide 5 description directly in plan.md
**Detected via:** git diff at start of Step 2
```
